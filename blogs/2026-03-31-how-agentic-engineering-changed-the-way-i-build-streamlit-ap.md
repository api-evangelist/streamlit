<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*Pj-wTC-bGtcbKWHqxpnhCw.png" /></figure><p>When you’re building tutorial content, good data is hard to come by. You need something real enough to tell a story, rich enough to be interesting, and relevant enough that developers actually care.</p><p>So when Joe Reis had just published his <a href="https://joereis.github.io/practical_data_data_eng_survey/">2026 State of Data Engineering</a> report (<em>i.e.</em> 1,136 survey responses from data engineering practitioners across the globe) I knew I had something worth building on. Columns on everything from orchestration tools to AI adoption to the bottlenecks keeping teams up at night.</p><p>I also had an idea: turn this static CSV into an interactive data exploration app where anyone could slice the data by role, industry, region, and org size. Not a single dashboard page, but a full multipage Streamlit app with distinct analysis angles spanning personas, tech stacks, AI adoption, organizational dynamics, pain points, education gaps, and practitioner voices.</p><p>A project like this would normally take a few months to build out. Back in 2023, I built the <a href="https://state-of-llm.streamlit.app/">State of LLM Apps 2023</a> in about 3 months.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*GW-Ad_u4kx6Et8FZtXVwaA.png" /></figure><p>With agentic engineering in Cortex Code, using custom agent skills, an AGENTS.md file with app design patterns, and an app.md planning doc for ideation, I built a similar app over the course of about a week, working in focused sessions between other tasks. That's 12 weeks compressed into a week! Here's how that played out.</p><h3>From vibe coding to agentic engineering</h3><p>But before we dive into the specifics, let’s take a moment to explore the shift from vibe coding to agentic engineering.</p><p>“AI-assisted coding” is the catch-all term for any coding activity that uses AI. It spans everything from vibe coding to agentic engineering and whatever comes next. Within that spectrum, the approaches differ sharply in how much control you hand over and how much structure you bring.</p><p>In early 2025, Andrej Karpathy coined <a href="https://x.com/karpathy/status/1886192184808149383">“vibe coding”</a> to describe the free-form, prompt-driven way people were generating software with AI: type a prompt, accept the output, “forget that the code even exists.” It captured the excitement of the moment. But for anything beyond a quick prototype, vibes alone produce fragile apps that need heavy rework. You end up spending more time debugging AI-generated code than you saved writing it.</p><p>A year later, Karpathy introduced a new term: <a href="https://x.com/karpathy/status/2019137879310836075">agentic engineering</a>. The distinction matters. “Agentic,” he writes, “because the new default is that you are not writing the code directly 99% of the time, you are orchestrating agents who do and acting as oversight. Engineering to emphasize that there is an art &amp; science and expertise to it. It’s something you can learn and become better at, with its own depth of a different kind.”</p><p>Simon Willison builds on this in his <a href="https://simonwillison.net/guides/agentic-engineering-patterns/what-is-agentic-engineering/">Agentic Engineering Patterns</a> guide: you define a goal, the coding agent generates and executes code in a loop until that goal is met, and your job is figuring out <em>what</em> to build, navigating tradeoffs, verifying results, and updating instructions so the agent learns from past mistakes. Vibe coding, Willison argues, is more useful in its original definition as a term for “unreviewed, prototype-quality LLM-generated code” rather than a catch-all for AI-assisted development.</p><p>This project was agentic engineering, not vibe coding.</p><p>Here’s how:</p><ul><li>The app.md planning doc defined the goals.</li><li>The AGENTS.md file and a custom streamlit-design-patterns skill provided the constraints and patterns.</li><li>I reviewed every page the agent produced, caught edge cases it missed, and refined the instructions for the next session.</li><li>The agent wrote the code.</li><li>I drove the architecture, the editorial decisions, and the quality bar.</li></ul><h3>Start with the story, not the code</h3><p>The instinct with vibe coding is to jump straight into prompting: “build me a dashboard.” I’ve learned the hard way that this produces generic apps that need heavy rework through several iterations.</p><p>Instead, I started by writing a planning document. Not code. A markdown file called app.md that laid out every page I wanted, what data each page needed, and the story each page should tell. I used the AI to help brainstorm page ideas and analysis angles, then curated and refined the best ones into the plan. The AI Paradox page, for example, needed to answer one question: <em>&quot;82% of practitioners use AI daily, but only 10% have it embedded organizationally. What explains the gap?&quot;</em></p><p>I spent nearly a day on planning alone. But every session after that was faster because of it.</p><h4>Excerpt from app.md (plan before code)</h4><p>The planning file lived in an app.md file and was written as a full build specification file (purpose, audience, repo layout, and navigation) before any implementation of the app. It opened like this:</p><pre># State of Data Engineering 2026 - Interactive Explorer App<br /><br />An app building plan for creating an interactive Streamlit application to explore the 2026 State of Data Engineering survey data, following BioCurator design patterns.<br /><br />---<br /><br />## 1. App Overview<br /><br />### Purpose<br />Transform static survey findings into an interactive exploration tool where users can discover insights relevant to their specific role, industry, and technology stack.<br /><br />### Target Users<br />- Data Engineers benchmarking their organization<br />- Data Leaders making strategic decisions<br />- Recruiters understanding the talent landscape<br />- Vendors understanding market adoption<br /><br />---<br /><br />## 2. App Structure<br /><br />state-of-engineering-app/<br />├── app.py                          # Main entry point (navigation + session state)<br />├── data_utils.py                   # Shared data loading, filtering, caching<br />├── chart_utils.py                  # Reusable chart components<br />├── pages/<br />│   ├── home.py                     # Overview dashboard with key stats<br />│   │<br />│   │   # Persona &amp; Benchmarking<br />│   ├── persona_matcher.py          # Find Your Tribe Persona Matcher<br />│   ├── regional_benchmark.py       # Regional Benchmark Tool<br />│   ├── solo_practitioner.py        # Solo Practitioner Experience<br />│   │<br />│   │   # Technology Stack &amp; Architecture  <br />│   ├── stack_explorer.py           # Technology Stack Explorer<br />│   ...                             # (additional page modules in the full doc)</pre><p>The full app.md went on to spell out navigation (st.navigation groupings), session state, sidebar filters, per-page specs, phased implementation order, and widget–chart pairings—so every later prompt could point at a single source of truth.</p><h3>Architecture: shared utilities, independent pages</h3><p>With the plan in hand, I asked Cortex Code to build the foundation first: a data_utils.py for loading and filtering survey data, and a chart_utils.py with reusable chart functions including donut charts, lollipop charts, diverging bars, Sankey diagrams, radar charts, and heatmaps.</p><p>This was a deliberate choice. Every page imports from the same two utility files, which means consistent styling, consistent data handling, and zero copy-paste drift across 17 pages. When I later tweaked the color palette from defaults to a custom rainbow scheme (#8B5CF6, #3B82F6, #10B981, #FACC15, #F59E0B, #EF4444), every chart across every page updated at once.</p><p>The main app.py handles navigation using st.navigation() with grouped sections (Persona, Tech Stack, AI Adoption, Organization, Pain Points, Skills) and a sidebar with global filters. Every page reads from get_filtered_df(), so when you filter to &quot;Finance, 1000+ employees&quot; in the sidebar, all 17 pages reflect that slice.</p><h3>Building pages with agent skills</h3><p>Once the foundation was solid, I started requesting pages in focused sessions, typically knocking out two or three pages per sitting. This is where the agentic engineering approach really paid off. I had a custom streamlit-design-patterns skill loaded into Cortex Code and an AGENTS.md file encoding app design patterns (layout conventions, charting preferences, Streamlit idioms). The agent didn't start from scratch each time; it worked from established patterns.</p><p>Each prompt followed a pattern:</p><p><em>Build the AI Paradox page. It should show personal AI usage vs organizational adoption as side-by-side donut charts, then filter to the “gap cohort” (daily AI users with low org adoption) and show their bottlenecks and industries. Add a select slider that lets users explore characteristics at each adoption level. End with a segment comparison that shows the personal-vs-org gap by industry, role, or region.</em></p><p>Cortex Code would generate the page, I’d review the output, and we’d iterate. The typical cycle was: generate, run, spot an issue, fix, move on. Most pages took a few rounds of iterations.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*IzDvJ149xVZY1j122ku9BQ.png" /></figure><p>Some patterns I noticed working well:</p><p><strong>Giving the AI the data schema up front.</strong> The survey had columns like ai_usage_frequency, ai_adoption, biggest_bottleneck, and modeling_pain_points (comma-separated multi-select). When I included these column names and their possible values in my prompts, the generated code was far more accurate.</p><p><strong>Requesting specific chart types.</strong> “Use a lollipop chart for bottlenecks” produced better results than “visualize the bottlenecks.” The AI knew exactly which chart_utils function to call.</p><p><strong>Adding insight captions programmatically.</strong> Each chart section ends with a dynamically generated st.caption() that highlights the top findings. For example: &quot;Legacy/Tech Debt leads at 24%, followed by Lack of leadership direction at 21%.&quot; These aren't hardcoded strings. They're dynamically computed from the filtered data, so they update as users apply filters.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*XD0sgDPt0QouWUMzaBnFMw.png" /></figure><h3>Tricky parts</h3><p>Not everything was smooth. Here’s a few lessons from the rough edges:</p><p><strong>Multi-select columns need preprocessing.</strong> The modeling_pain_points column contained comma-separated values like &quot;Pressure to move fast, Lack of ownership.&quot; I needed an explode_multiselect() utility to properly count and cross-tabulate these values. The AI initially tried to treat them as single categories.</p><p><strong>Global filter state management was fiddly.</strong> I wanted an “All Roles” option that auto-deselects when you pick specific roles, and auto-reselects when you clear everything. This required a custom init_multiselect_state() function that tracks previous selections via a shadow key in st.session_state. Getting this right took the most back-and-forth.</p><p><strong>Altair’s </strong><strong>labelLimit defaults truncate long labels.</strong> Survey responses like &quot;Lack of leadership direction / unclear business strategy&quot; would get cut off. Setting axis=alt.Axis(labelLimit=0) on every Y axis became a global pattern I enforced across all pages.</p><h3>What I’d do differently</h3><p>If I were starting over, I’d add the about.py page first instead of last. It provides context that helps users understand the data source, methodology, and limitations. Building it last meant I had to go back and verify all my assumptions about column meanings.</p><p>I’d also create a FILTERS.md or constants file defining all the ordered category lists (org sizes from &quot;&lt; 50 employees&quot; to &quot;10,000+&quot;, AI adoption from &quot;No meaningful adoption&quot; to &quot;AI embedded&quot;) in one place. I ended up defining these in data_utils.py, but having them as a separate reference would have made prompts cleaner.</p><h3>Result</h3><p>The final app has 17 interactive pages, 20 Python files, a shared color system, global sidebar filters, and dynamically generated insight text. Every chart responds to the global filters. The persona matcher computes similarity scores across four dimensions. The firefighting predictor cross-tabulates modeling approach, orchestration, and org size against firefighting rates.</p><p>Here’s what the completed app looked like:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/800/1*E99EB66HcWArIUlpSY94RQ.gif" /></figure><p>All from one CSV file, one planning document, and about a week of iterative prompting sessions using the agentic engineering workflow. Essentially, the provided prompts provided the intent that the agent skills used to guide it during the coding phase. This took us to a reasonably good starting point that required a few more iterations to fine-tune the app.</p><p>You can see the <a href="https://www.linkedin.com/feed/update/urn:li:ugcPost:7428522366877716480">LinkedIn video walkthrough</a> showing the app in action.</p><p><a href="https://www.linkedin.com/feed/update/urn:li:ugcPost:7428522366877716480/">2026 State of Data Engineering Survey Interactive Explorer | Chanin Nantasenamat posted on the topic | LinkedIn</a></p><h3>Takeaway</h3><p>Agentic engineering isn’t vibe coding. As Karpathy puts it, you’re orchestrating agents and acting as oversight, not hoping a single prompt produces something usable. As Willison describes, you define goals, verify results, and update your instructions so the agent improves over time. In practice, that meant giving the AI structured context to work from.</p><p>An app.md for ideation, an AGENTS.md for design patterns, custom skills for domain knowledge, and reusable utility modules for consistency. I handled the editorial decisions, including what story each page tells, what data supports it, and what the user should take away. The agent handled the execution, grounded in patterns I'd defined.</p><p>The planning document was worth more than any individual prompt. If you’re building something beyond a single-page prototype, invest in the context layer first. Your agent will build better software because of it.</p><h3>Conclusion</h3><p>The week-long build only worked because the unglamorous decisions (<em>i.e.</em> what to call each page, how files fit together, which chart library to standardize on) were settled in app.md before the first serious line of app code is written.</p><p>The product was not something the agent dreamed up, it was the implementation of a plan that I had already thought through on paper. Early on, a workflow like this can feel sluggish: you spend time on markdown, skills, and guardrails instead of visible UI. Treat that as front-loaded cost, not waste. What you buy is alignment: page seventeen still feels like the same app as page one, and a structure you can navigate and debug on your own, long after the session ends.</p><p>This is how I build Streamlit apps in the age of agentic AI. How do you build Streamlit apps?</p><p>Happy Streamlit-ing! 🎈</p><img alt="" height="1" src="https://medium.com/_/stat?event=post.clientViewed&amp;referrerSource=full_rss&amp;postId=27bb1c95134e" width="1" /><hr /><p><a href="https://blog.streamlit.io/how-agentic-engineering-changed-the-way-i-build-streamlit-apps-27bb1c95134e">How Agentic Engineering Changed the Way I Build Streamlit Apps</a> was originally published in <a href="https://blog.streamlit.io">Streamlit</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>
