<p>Features that used to take a sprint now ship in hours. Newly reported bugs are at a three-year low. The codebase is fixing its own flaky tests, updating docs, and closing coverage gaps — without anyone asking.</p><p>None of that happened because of clever prompts or waiting for the right model. It happened because we redesigned the codebase itself to work well for agents.</p><p><a href="https://github.com/streamlit/streamlit">Streamlit</a> is an open-source Python library used by over a million developers. Over the past year, we turned it into an environment where coding agents own the majority of feature work and maintenance while humans focus on product, design, and architecture decisions. The single biggest lesson: when an agent produces bad output, the problem is almost always the repo, not the AI.</p><p>This post is the playbook. We’ll walk through the layered system — rules, skills, subagents, hooks, and automation — that made it work, tuned for the tools our team actually uses: Claude Code and Cursor.</p><p>The foundation underneath all of it: the GitHub repo is the single source of truth — code, docs, specs, rules, and automation all live together in one place. An agent with access to the repo should be able to help with almost any project-related task.</p><h4>Swiss cheese, not silver bullets</h4><p>There is no single trick that makes agents reliable in a mature codebase. The mental model that has worked best for us is the Swiss-cheese model: many imperfect layers, stacked together.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*0dI1qG8bgra_vfS97Wo-Wg.png" /><figcaption>The Swiss cheese model to ensure high-quality output</figcaption></figure><p>Some layers prevent bad output: linting, type checking, tests, hooks. Some layers help agents produce good output in the first place: clear code, well-placed rules, architecture docs, specs. And some layers catch what slips through anyway: AI review, human review. No single slice is enough. The stack is what makes agents produce high-quality code.</p><p>These are the layers we focused on:</p><h4>1. Simplify until agents stop tripping</h4><p>Humans can adapt to friction over time. We learn the weird build step, the inconsistent naming, and the undocumented workflow. Agents have a much lower tolerance for ambiguity, inconsistency, and unnecessary complexity.</p><p>So we started treating agent confusion as signal. If agents repeatedly got lost in one area of the codebase, produced bad code, or slowed down on a recurring workflow, that was usually a sign that the system itself needed work.</p><p>We refactored, modernized, and simplified parts of the codebase and the development setup to remove tech debt and ambiguity. We also sped up key workflows, because fast feedback loops matter a lot when an agent is doing the work.</p><p>This turned out to be one of the most useful mindset shifts for us: where agents struggle is often where the codebase wants simplification anyway.</p><h4>2. Install hard guardrails and make them fast</h4><p>If you want more feature velocity in a widely used library, you need stronger guardrails, not weaker ones. Otherwise, you just move the bottleneck from implementation to review and cleanup.</p><p>So we tightened linting, formatting, type checking, and pre-commit enforcement across Python, TypeScript, and Protobuf — the stack we use in Streamlit — with automated fixes wherever possible.</p><p>On the Python side, that meant roughly 800 <em>ruff</em> rules (ALL + preview), <em>ty</em> as an ultra-fast type checker to keep the feedback loop fast, and <em>mypy</em> strict mode across all Python files. On the TypeScript side, that meant stricter <em>eslint</em> and type-checking rules, custom rules for project-specific constraints, <em>oxlint</em> for fast agent feedback, and <em>knip</em> to keep the project clean. For Protobuf, we added <em>buf</em> for linting and formatting.</p><p>Beyond individual tools, we added a fast <a href="https://github.com/streamlit/streamlit/blob/develop/Makefile">make check</a> path that runs the relevant checks and tests for the files changed in a branch. On top of that, we added CI checks and dashboards around <a href="https://issues.streamlit.app/wheel_size)">size</a>, <a href="https://issues.streamlit.app/playwright">performance</a>, <a href="https://issues.streamlit.app/Test_Coverage_(Python)">coverage</a>, <a href="https://issues.streamlit.app/flaky_tests">flaky tests</a>, <a href="https://issues.streamlit.app/load_testing">load tests</a>, and <a href="https://issues.streamlit.app/playwright_stats">e2e health</a>.</p><p>We encoded policy into hooks so agents do not need to remember everything perfectly, e.g.:</p><pre>.claude/hooks/<br />├── pre_bash_redirect.py # block non-compliant commands, suggest the right ones<br />├── post_edit_autofix.sh # auto-fix files after edits<br />└── stop_check.sh # run make check before completion</pre><p>The point was not process for the sake of process. The point was to make the correct path cheap, fast, visible, and enforceable.</p><h4>3. Move context into the repo</h4><p>Agents can only reliably use what they can actually discover. So we kept pushing project knowledge into version control.</p><p>We added <a href="https://github.com/search?q=path%3A**%2FAGENTS.md+repo%3Astreamlit%2Fstreamlit+&amp;type=code&amp;ref=advsearch">AGENTS.md files throughout the codebase</a> and kept refining them whenever agents ran into friction or repeated mistakes. Those files live close to the code they describe, so the guidance is local instead of buried in some global prompt.</p><p>We cleaned up noisy logs across development commands and added more useful error output. Agents read logs. Bad logs mean bad context.</p><p>We also moved more durable knowledge into the repo: <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/understanding-streamlit-architecture/SKILL.md">architecture docs</a>, <a href="https://github.com/streamlit/streamlit/blob/develop/specs/AGENTS.md#principles-of-streamlit-api-design">API design principles</a>, the <a href="https://github.com/streamlit/streamlit/tree/develop/wiki)">development wiki</a>, and our <a href="https://github.com/streamlit/streamlit/tree/develop/specs">speccing process in specs/</a>. Specs are our primary artifact for new features — a practice known as spec-driven development. A well-written spec gives an agent enough context to produce a working implementation, and gives reviewers a clear reference point.</p><p>We keep coming back to the same rule: if knowledge matters repeatedly, it should not stay trapped in a review comment, a Notion page, a Jira ticket, or somebody’s head. It belongs in the repo.</p><p>Every PR review is a context audit. Repeated feedback is often just missing repository knowledge in disguise. The exact way different tools consume that context will keep changing, and that is fine. What matters is that the rules, docs, specs, and workflow definitions live in version control instead of in some isolated space.</p><p>Not all agent-generated content belongs in the main repo, though. Implementation plans, codebase explorations, and design notes are valuable during review but are not long-lived documentation. We needed a scratch space for agents — somewhere to put working material that reviewers can find but that does not pollute the main repo.</p><p>We ended up repurposing the <a href="https://docs.github.com/en/communities/documenting-your-project-with-wikis/about-wikis">GitHub wiki git repo</a> for this. Every GitHub repository comes with a wiki repo attached to it. We disabled the wiki UI and use the bare git repo purely as a shared artifact store. Agents push documents there organized by PR number and link them automatically so reviewers can see the full context. Since the wiki repo is checked out locally, agents working on new changes also have access to prior implementation plans and explorations, which helps them build on existing work.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*GZpMtT7zpCVBxpODp9w2Iw.png" /><figcaption>Automated triage comment on a newly filed issue</figcaption></figure><p>It is a hack, honestly. But it works surprisingly well as a lightweight, version-controlled scratch space that lives right next to the main repo.</p><h4>4. Turn recurring work into reusable skills</h4><p>Once the basics were in place, we started packaging recurring engineering work into reusable skills and subagents. A lot of “agent productivity” is just taking messy recurring work and turning it into explicit, well-defined workflows.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*PkJm173ushVmw0yn7ptK6Q.png" /><figcaption>Skill and subagent stack in the Streamlit repo</figcaption></figure><p>At the bottom are atomic skills: small, focused capabilities that do one thing well. Examples include <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/checking-changes/SKILL.md">checking-changes</a>, <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/debugging-streamlit/SKILL.md">debugging-streamlit</a> <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/addressing-pr-review-comments/SKILL.md">addressing-pr-review-comments</a>, and <a href="https://github.com/streamlit/streamlit/tree/develop/.claude/skills/updating-internal-docs">updating-internal-docs</a>.</p><p>Above that are composite subagents like <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/agents/simplifying-local-changes.md">simplifying-local-changes</a>, <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/agents/reviewing-local-changes.md">reviewing-local-changes</a>, and <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/agents/fixing-pr.md">fixing-pr</a>. We use subagents when a workflow benefits from an isolated context, so higher-level tasks can run end-to-end without dragging around a giant context window.</p><p>At the top are orchestrator skills like <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/finalizing-pr/SKILL.md">finalizing-pr</a> and <a href="https://github.com/streamlit/streamlit/blob/develop/.claude/skills/implementing-feature/SKILL.md">implementing-feature</a>, which take a broader goal and coordinate the lower-level pieces into a full workflow.</p><p>A skill is more than a prompt: it is an engineered, reusable unit of process that can be composed into higher-level workflows.</p><p>One thing we still lack is good visibility into how well this system works — which skills get invoked, what documentation gets read, where agents slow down. We added basic invocation metrics to PRs, but there is room to improve.</p><h4>5. Let AI review AI</h4><p>As output goes up, review becomes the next bottleneck. So we invested in multiple layers of AI review instead of expecting human reviewers to absorb all of the extra volume.</p><p>On the issue side, we added a <a href="https://github.com/streamlit/streamlit/blob/develop/.github/workflows/ai-issue-triage.yml">custom AI issue triage workflow</a> that provides an initial analysis, finds related issues, and applies labels. On the PR side, we use GitHub Copilot, Cursor BugBot, and Graphite as a first layer of fast automated review.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*txI7hofBRiSlhl14JXDXKg.png" /><figcaption>Multi-model PR review</figcaption></figure><p>We also built a <a href="https://github.com/streamlit/streamlit/blob/develop/.github/workflows/ai-pr-review.yml">custom multi-model PR review workflow</a>. It lets us review a PR with an arbitrary combination of models. Right now that includes <em>gpt-5.3-codex-high</em>, <em>gemini-3.1-pro</em>, and <em>opus-4.6-thinking</em>, with one acting as the judge.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*HrSG-qfb7syGw-kwzTwUHg.png" /><figcaption>Multi-model PR review comment</figcaption></figure><p>That review layer has been one of the highest-leverage pieces of the whole setup. It catches non-obvious issues, and by the time a human reviewer opens the PR, most of the boring stuff is already handled.</p><h4>6. Automate everything</h4><p>Once a workflow is stable, manual repetition becomes hard to justify. So we automated the recurring work: dependency updates, parts of the release process, asset updates, cleanups, and a growing set of AI-powered GitHub Actions workflows. A few examples:</p><p>- <a href="https://github.com/streamlit/streamlit/blob/develop/.github/workflows/ai-update-docs.yml">ai-update-docs</a> runs every week to keep internal docs aligned with the current codebase [<a href="https://github.com/streamlit/streamlit/pull/14493">example</a>].<br />- <a href="https://github.com/streamlit/streamlit/blob/develop/.github/workflows/ai-fix-flaky-e2e-tests.yml">ai-fix-flaky-e2e-tests</a> runs every week to detect and fix e2e flakiness [e<a href="https://github.com/streamlit/streamlit/pull/14624">xample</a>].<br />- <a href="https://github.com/streamlit/streamlit/blob/develop/.github/workflows/ai-test-coverage.yml">ai-test-coverage</a> automatically improves unit test coverage.</p><p>As our confidence in the harness increases, we are gradually moving more tasks into automated workflows: implement a feature PR when a spec is merged, implement bug-fix PRs when issues arrive, autofix PRs when CI fails or review comments come in, and automate more of the maintenance layer.</p><h4>How our work changed</h4><p>When we started, most of the team’s time went into feature implementation, issue triage, bug fixes, and maintenance. Now, agents handle the majority of that work. The team spends more time on product decisions, architecture, design, and review — the work that benefits most from human judgment.</p><p>That shift did not come from better prompts or models. It came from optimizing the repo itself to work well for agents — layer by layer. Once code, docs, specs, rules, workflows, and automation all live together, the agent starts to behave like a productive contributor operating inside a well-designed system.</p><p>We are not done. There is always more context to move into the repo, more workflows to automate, more layers to stack. Every improvement to the harness compounds into higher velocity.</p><img alt="" height="1" src="https://medium.com/_/stat?event=post.clientViewed&amp;referrerSource=full_rss&amp;postId=75629a953354" width="1" /><hr /><p><a href="https://blog.streamlit.io/the-repo-is-the-harness-how-we-made-an-8-year-old-codebase-agent-native-75629a953354">The Repo Is the Harness — How we made an 8-year-old codebase agent-native</a> was originally published in <a href="https://blog.streamlit.io">Streamlit</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>
