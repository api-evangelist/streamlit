<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*LBorob3fim-OfFGch8NJBw.png" /><figcaption>Image generated by Author using the ideogram-ai/ideogram-v2-turbo model on Replicate.</figcaption></figure><p>Vibe coding has changed how we build apps. You describe what you want, and after a few seconds, AI generates the code. After a few rounds of iteration and you have a working Streamlit dashboard app. It’s genuinely magical.</p><p>But here’s the catch, without understanding how Streamlit actually works under the hood, that rapid progress can hit a wall. Your app suddenly behaves strangely. Data disappears. Buttons don’t seem to work. Performance grinds to a halt.</p><p>After spending considerable time on the<a href="https://discuss.streamlit.io/"> Streamlit Community Forum</a> and reviewing countless questions from beginners (some of whom were vibe coding their first apps), I started noticing patterns. The same confusions kept surfacing. The same “why isn’t this working?!” moments.</p><p>The good news? Most of these issues stem from a handful of core misunderstandings about how Streamlit actually works under the hood. Once you internalize these concepts, the framework clicks into place and you can vibe code with confidence instead of crossing your fingers every time you hit save.</p><p>Rather than cataloging every possible mistake (that list would be quite extensive), I’ve distilled these into <strong>10 core principles</strong>. Master these, and you’re on your way to building Streamlit apps with confidence.</p><p>Let’s dive in!</p><h3>1. Understand that your script reruns completely on every interaction</h3><p>This is the concept. The one that explains 80% of the unexpected behavior that you’ll encounter. If you only internalize one thing from this entire post, make it this:</p><p>Every button click, slider move, or text input triggers a complete rerun of your script from top to bottom. Not a partial update. Not a diff. The whole thing, starting from import streamlit as st and all the way down.</p><pre># ❌ This resets to 0 on every interaction<br />counter = 0<br />if st.button(&quot;Add&quot;):<br />    counter += 1<br />st.write(counter)  # Always shows 0 or 1<br /><br /># ✅ Use session state for persistence<br />if &quot;counter&quot; not in st.session_state:<br />    st.session_state.counter = 0<br />if st.button(&quot;Add&quot;):<br />    st.session_state.counter += 1<br />st.write(st.session_state.counter)  # Accumulates correctly</pre><p>Here are some common symptoms that we may see if not understanding this concept:</p><ul><li>Variables “reset” unexpectedly</li><li>Buttons seem to “not work”</li><li>Data disappears after clicking something else</li><li>Chat history vanishes</li></ul><p><strong>The fix: </strong><strong>st.session_state</strong></p><p>Session state is a dictionary that persists across reruns for each user session. The key pattern to memorize:</p><pre># Always initialize before using<br />if &quot;counter&quot; not in st.session_state:<br />    st.session_state.counter = 0</pre><p>This check ensures you only set the default once (on the first run), and subsequent reruns use the existing value. Skip this check, and you’ll reset your variable on every interaction.</p><p>When to use session state:</p><ul><li>Counters, toggles, and flags</li><li>User inputs you need to preserve</li><li>Chat message history</li><li>Multi-step form data</li><li>Any value that should survive a rerun</li></ul><p>Once this pattern becomes second nature, most “mysterious” Streamlit behavior suddenly makes perfect sense.</p><h3>2. Cache expensive operations, always</h3><p>Remember that rerun behavior from Principle 1? Here’s where it becomes a performance nightmare if you’re not careful.</p><p>If your code takes time to run (loading files, API calls, ML models), it will run on <em>every single interaction</em> unless you cache it. Your users don’t want to wait 10 seconds every time they adjust a slider.</p><pre># ❌ Loads 500MB file on every click<br />data = pd.read_csv(&quot;large_file.csv&quot;)<br /><br /># ✅ Cache it - runs once, reuses result<br />@st.cache_data<br />def load_data():<br />    return pd.read_csv(&quot;large_file.csv&quot;)<br /><br />data = load_data()</pre><p><strong>The two decorators:</strong></p><ul><li>@st.cache_data: For data (DataFrames, lists, API responses). Returns a copy.</li><li>@st.cache_resource: For resources (DB connections, ML models). Returns the same object.</li></ul><p>Add TTL for data that updates frequently:</p><pre>@st.cache_data(ttl=300)  # Refresh every 5 minutes<br />def get_live_data():<br />    return api.fetch()</pre><p>Why does this work? The decorator stores the function’s return value in memory, keyed by the function’s input arguments. On subsequent calls with the same arguments, Streamlit returns the cached result instead of re-executing the function. This means your 10-second data load happens only once, on that initial run, and not on every interaction.</p><h3>3. Never hardcode secrets: use st.secrets</h3><p>This one isn’t about convenience. It’s about not accidentally publishing your API keys to GitHub for the world to see.</p><p>Yes, bots can actively scan for this. And yes, your keys will get compromised within minutes.</p><p>This principle has no exceptions. “Just for testing” is how secrets can end up in the commit history forever.</p><pre># ❌ NEVER do this<br />api_key = &quot;sk-1234567890&quot;<br /><br /># ✅ Use .streamlit/secrets.toml (with gitignored!)<br />api_key = st.secrets[&quot;openai&quot;][&quot;api_key&quot;]</pre><p>Here’s what to do instead:</p><ul><li>Create .streamlit/secrets.toml for local development</li><li>Add .streamlit/secrets.toml to .gitignore before your first commit</li><li>Add secrets via the UI in the secrets management when deploying to Community Cloud</li></ul><h3>4. Handle empty and error states gracefully</h3><p>Your app will encounter None values. It will receive empty lists. APIs will fail. Files won’t upload.</p><p>The difference between a frustrating app and a delightful one is often just how gracefully it handles these situations. Never assume that data exists; always check before processing.</p><pre># ❌ Crashes if no file was uploaded<br />uploaded = st.file_uploader(&quot;Upload&quot;)<br />df = pd.read_csv(uploaded)<br /><br /># ✅ Check first, guide the user<br />uploaded = st.file_uploader(&quot;Upload&quot;)<br />if uploaded is not None:<br />    df = pd.read_csv(uploaded)<br />    st.dataframe(df)<br />else:<br />    st.info(&quot;👆 Upload a CSV to get started&quot;)</pre><p>Apply this pattern everywhere:</p><ul><li>File uploads → check with is not None</li><li>API calls → wrap in try / except</li><li>Filtered data → check if result is empty</li><li>Selectbox options → verify that list isn’t empty</li></ul><p>Here’s an example for select boxes:</p><pre># Empty options can crash selectbox<br />options = [x for x in items if x.startswith(&quot;A&quot;)]<br />if options:<br />    selected = st.selectbox(&quot;Choose&quot;, options)<br />else:<br />    st.warning(&quot;No matching items found&quot;)</pre><p>Why does this matters? Streamlit apps often start in an “empty” state before the user provides input. Your code needs to handle this gracefully. Think of it as defensive coding: assume nothing exists until you’ve verified that it does.</p><h3>5. Use forms when you need to batch inputs</h3><p>Here’s a scenario: you have a form with name, email, and phone fields. Without Streamlit forms, your app might start processing (triggering a rerun of the app) after the user starts typing their name, before they’ve even gotten to their email. As a result, the app may rerun multiple times by the time that you’re finished with filling in the input fields.</p><p>Forms let users fill out multiple fields and submit them together as a batch, thereby giving you control over when the processing actually happens.</p><pre># ❌ Processing might trigger on partial input<br />name = st.text_input(&quot;Name&quot;)<br />email = st.text_input(&quot;Email&quot;)<br /><br />if name:<br />    process1(name)<br />if email:<br />    process2(email)<br /><br /># ✅ Forms batch until explicit submit<br />with st.form(&quot;user_form&quot;):<br />    name = st.text_input(&quot;Name&quot;)<br />    email = st.text_input(&quot;Email&quot;)<br />    submitted = st.form_submit_button(&quot;Submit&quot;)<br /><br />if submitted:<br />    if name and email:  # Only process if both are filled<br />        process1(name)<br />        process2(email)<br />    else:<br />        st.error(&quot;Please fill out all fields&quot;)</pre><p>Use forms when:</p><ul><li>You have multiple related inputs</li><li>You want to prevent premature processing</li><li>You’re building data entry interfaces</li></ul><p>Forms create a boundary that prevents reruns until the user explicitly clicks submit. All widgets inside the form collect input without triggering the script, then submit sends everything at once. This gives you a clean “before” and “after” moment instead of continuous partial updates.</p><h3>6. Add unique keys to widgets in loops</h3><p>DuplicateWidgetID is one of those error messages that makes perfect sense once you understand it, and is completely baffling before that moment.</p><p>When creating widgets dynamically (in loops or conditionally), Streamlit needs unique identifiers to tell them apart. No key? Streamlit guesses. Multiple widgets with the same guess? Chaos.</p><pre># ❌ DuplicateWidgetID error<br />for item in items:<br />    st.checkbox(item)<br /><br /># ✅ Add unique keys<br />for i, item in enumerate(items):<br />    st.checkbox(item, key=f&quot;checkbox_{i}&quot;)</pre><p>Also remember:</p><ul><li>Capture widget return values: value=st.slider(&quot;...&quot;)</li><li>Widgets with keys automatically sync to st.session_state</li></ul><p>Let’s take a look at an example:</p><pre># ✅ Captures the return value from a stored variable<br />temperature = st.slider(&quot;Temperature&quot;, 0, 100, 72)<br />st.write(f&quot;Current: {temperature}°F&quot;)<br /><br /># ✅ A key is automatically stored in session_state<br />st.slider(&quot;Humidity&quot;, 0, 100, 50, key=&quot;humidity&quot;)<br />st.write(f&quot;Humidity: {st.session_state.humidity}%&quot;)</pre><p>Direct capture: The slider returns its current value, which we store in the temperature variable. This is the simplest approach when you just need to use the value immediately.</p><p>Using a key: When you add a key parameter, Streamlit automatically stores the widget’s value in st.session_state under that key name. You can access it as st.session_state.humidity anywhere in your app. This is useful when you need to access the value in multiple places or modify it programmatically.</p><p>Streamlit identifies widgets by generating an ID from their type, label, and position in the code. In a loop, multiple widgets can end up with identical IDs and thus give rise to an error! The key parameter overrides this auto-generated ID with your own unique identifier, letting Streamlit distinguish between them. As a bonus, any widget with a key automatically syncs its value to st.session_state, making it accessible throughout your app.</p><h3>7. Show progress for long operations</h3><p>A frozen screen is a source of anxiety for users. “Did it crash? Should I refresh? Is my data lost?” These are not the questions you want people asking.</p><p>The fix is simple: show them something is happening. Streamlit makes this almost very easy.</p><pre># ❌ User sees frozen screen, thinks app crashed<br />result = slow_api_call()<br />st.write(result)<br /><br /># ✅ Show a spinner for single operations<br />with st.spinner(&quot;Generating response...&quot;):<br />    result = slow_api_call()<br />st.write(result)<br /><br /># ✅ Use st.status for multi-step operations<br />with st.status(&quot;Processing...&quot;, expanded=True) as status:<br />    st.write(&quot;Step 1: Loading data...&quot;)<br />    data = load_data()<br />    <br />    st.write(&quot;Step 2: Analyzing...&quot;)<br />    analysis = analyze(data)<br />    <br />    st.write(&quot;Step 3: Generating report...&quot;)<br />    report = generate_report(analysis)<br />    <br />    status.update(label=&quot;Complete!&quot;, state=&quot;complete&quot;)</pre><p>For LLM apps, streaming responses can provide users with real-time feedback.</p><p>Here’s how we could implement streaming using OpenAI:</p><pre>from openai import OpenAI<br />import streamlit as st<br /><br />client = OpenAI(api_key=st.secrets[&quot;openai&quot;][&quot;api_key&quot;])<br /><br />stream = client.chat.completions.create(<br />    model=&quot;gpt-4o&quot;,<br />    max_tokens=1024,<br />    messages=[{&quot;role&quot;: &quot;user&quot;, &quot;content&quot;: &quot;Write a poem about snowflakes.&quot;}],<br />    stream=True<br />)<br /><br />st.write_stream(stream)  # Shows tokens as they arrive</pre><p>And here’s how with Anthropic:</p><pre>from anthropic import Anthropic<br />import streamlit as st<br /><br />client = Anthropic(api_key=st.secrets[&quot;anthropic&quot;][&quot;api_key&quot;])<br /><br /># Stream responses for real-time feedback<br />with client.messages.stream(<br />    model=&quot;claude-sonnet-4-5&quot;,<br />    max_tokens=1024,<br />    messages=[{&quot;role&quot;: &quot;user&quot;, &quot;content&quot;: &quot;Write a poem about snowflakes.&quot;}]<br />) as stream:<br />    st.write_stream(stream.text_stream)  # Shows tokens as they arrive</pre><p>Why does this matter? Users are surprisingly patient when they can see progress. A 10-second wait with a spinner feels faster than a 5-second frozen screen. It’s not about speed, it’s about communication.</p><h3>8. Use @st.fragment for partial reruns</h3><p>This is the “I wish I’d known this sooner” feature for anyone building apps with both expensive visualizations and interactive elements.</p><p>By default, the entire script reruns on every interaction. Got a chart that takes 5 seconds to render? Every widget interaction would cause the user to wait for another 5 more seconds as the app reruns. Fragments let you break this cycle by isolating sections that should rerun independently from the rest.</p><pre># ❌ Expensive operation reruns on every interaction<br />st.title(&quot;Dashboard&quot;)<br /><br /># This takes 5 seconds<br />plot_expensive_chart(data)<br /><br /># Every button click reruns the expensive chart!<br />if st.button(&quot;Increment Counter&quot;):<br />    st.session_state.count = st.session_state.get(&quot;count&quot;, 0) + 1<br />st.write(f&quot;Count: {st.session_state.get('count', 0)}&quot;)<br /><br /># ✅ Fragment isolates the counter from the expensive chart<br />st.title(&quot;Dashboard&quot;)<br /><br />plot_expensive_chart(data)  # Only runs on full page rerun<br /><br />@st.fragment<br />def counter_section():<br />    if st.button(&quot;Increment Counter&quot;):<br />        st.session_state.count = st.session_state.get(&quot;count&quot;, 0) + 1<br />    st.write(f&quot;Count: {st.session_state.get('count', 0)}&quot;)<br /><br />counter_section()  # This section reruns independently</pre><p>Use fragments when:</p><ul><li>Mixing expensive calculations with lightweight interactions</li><li>Any section that updates frequently while others stay static</li></ul><p>Why this works: The @st.fragment decorator creates an isolated execution scope that allows only a specific function to re-execute, not the entire script. Everything outside the fragment stays untouched, thereby preserving your expensive computations.</p><h3>9. Structure your project for deployment from the start</h3><p>“I’ll organize it later” is the lie we tell ourselves right before our app.py hits 2,000 lines and we can’t find anything.</p><p>Future you will thank present you for taking 5 minutes to set up a proper project structure. Don’t wait until deployment to organize your code.</p><pre>my_app/<br />├── app.py                 # Entry point<br />├── requirements.txt       # Pin versions!<br />├── .gitignore            # Include secrets.toml<br />├── .streamlit/<br />│   ├── config.toml       # Theme settings<br />│   └── secrets.toml      # Local secrets (gitignored)<br />├── pages/                # Multipage app<br />│   └── 1_📊_Dashboard.py<br />└── utils/                # Shared code<br />    └── data.py</pre><p>Why pin versions? An unpinned requirements.txt is a ticking time bomb. Your app works perfectly today, but six months from now when you redeploy or reinstall dependencies, a new library version might introduce breaking changes. Pinning versions (pandas==2.1.0 instead of just pandas) ensures your app behaves the same way every time (like putting your app in a time capsule).</p><p>Deployment checklist:</p><ul><li>requirements.txt with pinned versions (pandas==2.1.0)</li><li>Relative file paths only (use assets/ as oppossed to /Users/chanin/dashboard_app/assets)</li><li>Secrets in secrets.toml, not hard-coded in the code</li><li>Test your deployment early and often</li></ul><h3>10. Use Streamlit’s built-in features before custom solutions</h3><p>Before you spend an hour crafting custom CSS for a metrics display, check if Streamlit already has a widget for that.</p><p>It probably does and LLMs may be trained on an older version of Streamlit</p><p>The framework has grown significantly, and many common UI patterns now have first-class support. Using built-ins means less code to maintain, better performance, and a more consistent look.</p><pre>+----------------------+------------------------------------------+<br />| Need                 | Use This                                 |<br />+----------------------+------------------------------------------+<br />| Show loading state   | with st.spinner(&quot;Loading...&quot;)            |<br />| Display KPIs         | st.metric(label, value, delta)           |<br />| Organize content     | st.columns(), st.tabs(), st.expander()   |<br />| Database connections | st.connection()                          |<br />| Long operations      | st.status() for multi-step progress      |<br />| Chat interfaces      | st.chat_message(), st.chat_input()       |<br />| Stream LLM responses | st.write_stream()                        |<br />| Partial reruns       | @st.fragment                             |<br />+----------------------+------------------------------------------+</pre><p>Let’s say that you want to display metrics in your dashboard app, instead of manually using st.write() you could instead leverage st.metric(), which comes with the metric name and values as well as any recent positive/negative changes.</p><pre># ❌ Manual KPI formatting<br />st.write(f&quot;Revenue: ${revenue:,}&quot;)<br />st.write(f&quot;Change: {change:+.1%}&quot;)<br /><br /># ✅ Use st.metric<br />st.metric(&quot;Revenue&quot;, f&quot;${revenue:,}&quot;, f&quot;{change:+.1%}&quot;)</pre><p>If Streamlit doesn’t have it built-in, check these before building custom solutions:</p><ol><li><a href="https://extras.streamlit.app/"><strong>Streamlit Extras</strong></a>: A curated collection of additional components including card layouts, annotated text, clickable images, and more. Maintained by members of the Streamlit data science team.</li><li><a href="https://streamlit.io/components"><strong>Streamlit Components Gallery</strong></a>: Third-party extensions for bringing in additional functionality to your Streamlit app!</li></ol><p>The hierarchy: <strong>Built-in → Streamlit Extras → Third-party → Custom code</strong>. Each step down means more maintenance burden, so only go custom when you truly need something unique.</p><h3>Wrapping Up</h3><p>There you have it, 10 principles that address the root causes of most Streamlit beginner struggles. The beautiful thing about these principles is that once they click, they become second nature. You’ll start writing Streamlit code that “just works” because you understand <em>why</em> it works.</p><p>Vibe coding isn’t about blindly accepting AI suggestions. It’s about knowing enough to guide the conversation productively. The AI handles the syntax; you bring the architectural intuition.</p><p>If you’re looking to put these principles into practice, check out the upcoming <a href="https://30daysofai.streamlit.app/"><strong>30 Days of AI</strong></a><strong> </strong>in January 2026.
